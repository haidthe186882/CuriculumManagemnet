Different actors often correspond to different user classes, or roles, identified from the customer community that will use the product. Name the primary actor that will be initiating this use case and any other secondary actors who will participate in completing execution of the use case.

Description
Provide a brief description of the reason for and outcome of this use case, or a high-level description of the sequence of actions and the outcome of executing the use case.
Preconditions
List any activities that must take place, or any conditions that must be true, before the use case can be started. The system must be able to test each precondition. Number each precondition. Example: PRE-1: User’s identity has been authenticated.
Postconditions
Describe the state of the system at the successful conclusion of the use case execution. Label each postcondition in the form POST-X, where X is a sequence number. Example: POST-1: Price of item in the database has been updated with the new value.
Normal Flow
Provide a description of the user actions and corresponding system responses that will take place during execution of the use case under normal, expected conditions. This dialog sequence will ultimately lead to accomplishing the goal stated in the use case name and description. Document other successful usage scenarios that can take place within this use case. State the alternative flow, and describe any differences in the sequence of steps that take place. Number each alternative flow in the form “X.Y”, where “X” is the Use Case ID and Y is a sequence number for the alternative flow. For example, “5.3” would indicate the third alternative flow for use case number 5. Indicate where each alternative flow would branch off from the normal flow, and if pertinent, where it would rejoin the normal flow.
Exceptions
Describe any anticipated error conditions that could occur during execution of the use case and how the system is to respond to those conditions. Priority
Indicate the relative priority of implementing the functionality required to allow this use case to be executed. Use the same priority scheme as that used for the functional requirements.
Frequency of Use
Estimate the number of times this use case will be performed per some appropriate unit of time. This gives an early indicator of throughput, concurrent usage loads, and transaction capacity.
Business Rules

Other Information

Describe what should happen if the use case execution fails for some unanticipated or systemic reason (e.g., loss of network connectivity, timeout). 4.2 Quality Attributes
4.2.1 Usability
The FLM system shall be easy to learn and operate for users with different roles.
Requirements:
Learnability
A new student shall be able to search and view learning materials within 5 minutes without training.
A lecturer shall be able to upload learning materials within 10 minutes after first login.
A reviewer shall be able to review and approve a syllabus within 15 minutes after training.
User Interface Consistency
All pages shall follow a consistent navigation structure.
Buttons, forms, and data tables shall maintain consistent layouts across modules.
Accessibility
The system shall support keyboard navigation.
The system shall provide readable font sizes.
The system shall comply with WCAG 2.1 Level AA guidelines where feasible.
Error Prevention
Mandatory fields shall be clearly marked.
Validation messages shall be displayed before submission.
Users shall receive meaningful error messages when operations fail.
User Satisfaction
At least 90% of surveyed users shall successfully complete common tasks without assistance.

Resource Utilization
Under normal load:
CPU utilization shall remain below 70%.
Memory utilization shall remain below 80%.
Database connection pool utilization shall remain below 75%.
4.2.3 Reliability
The system shall provide stable and dependable operation.
Requirements:
System availability shall be at least 99.5%.
No single transaction shall result in data corruption.
All critical operations shall use database transactions.
Automatic recovery shall occur after unexpected failures.
Backup requirements:
Daily database backup.
Weekly full backup.
Recovery Point Objective (RPO) ≤ 24 hours.
Recovery Time Objective (RTO) ≤ 4 hours.
4.2.4 Security
The system shall protect academic data and user information.
Authentication:
Users shall authenticate through FPT authentication services.
Passwords shall never be stored in plain text.
Authorization:
Role-Based Access Control (RBAC) shall be implemented.
Permissions shall be enforced at both UI and API levels.
Data Protection:
All communications shall use HTTPS.
Sensitive information shall be encrypted in transit.
Audit logs shall record all critical modifications.
Session Management:
Session timeout shall occur after 30 minutes of inactivity.
Multiple failed login attempts shall trigger temporary account lockout.
4.2.5 Maintainability
The system shall be easy to modify and maintain.
Requirements:
The system shall follow a layered architecture.
Business logic shall be separated from presentation logic.
Database schema shall satisfy Third Normal Form (3NF).
Source code shall follow coding standards and naming conventions.
New modules shall be integrated without affecting existing modules.
4.2.6 Scalability
The system shall support future growth.
Requirements:
The system shall support horizontal application scaling.
Database performance shall remain acceptable when data volume doubles.
Additional campuses and academic programs shall be added without schema redesign.
4.2.7 Availability
The system shall remain accessible during academic operations.
Requirements:
System uptime shall be at least 99.5%.
Planned maintenance shall be announced at least 24 hours in advance.
Critical services shall be monitored continuously.
4.2.8 Auditability
The system shall maintain complete audit trails.
Requirements:
All create, update, delete operations shall be logged.
Workflow approvals and rejections shall be traceable.
Login history shall be retained for at least 12 months.
Audit logs shall not be editable by normal users.
4.2.9 Data Integrity
The system shall ensure consistency and correctness of data.
Requirements:
Foreign key constraints shall be enforced.
Duplicate subject codes shall not be allowed.
Duplicate curriculum codes shall not be allowed.
Version histories shall be preserved for syllabus modifications.
Transaction rollback shall occur upon failure.
